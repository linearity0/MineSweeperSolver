## 一、项目背景与选题动机

扫雷（Minesweeper）是一款经典的逻辑推理游戏。玩家根据已翻开格子的数字（表示周围 8 格中雷的个数），推断哪些未翻开格子是雷、哪些是安全的。

初级难度（9×9，10 雷）可凭直觉完成，但中级（16×16，40 雷）和高级（16×30，99 雷）经常出现需要**跨多个数字格联合推理**的情况——这正是数据结构与算法课程所学知识的应用场景。

本项目的核心问题：**如何用程序模拟人类的扫雷推理过程？确定性规则能解多少局面？无法确定时如何处理？**

## 二、总体设计

### 2.1 架构

```
GUI（pygame）
  │
  ▼
求解器入口（solver.py）
  │
  ├── 第一层：单点推理  ──→ 确定性规则匹配
  ├── 第二层：约束传播  ──→ 集合子集推导
  └── 第三层：概率评估  ──→ 蒙特卡洛采样
  │
  ▼
游戏引擎（engine.py）
  棋盘状态管理、BFS 扩散、胜负判定
```

逻辑与界面分离：`engine.py` 和 `ai/` 只处理数据与算法，`gui.py` 只负责渲染和交互。

### 2.2 数据结构选型

| 数据 | 选用结构 | 原因 |
|------|---------|------|
| 棋盘 | 二维 `list[list[Cell]]` | O(1) 随机访问，邻域遍历直观 |
| BFS 队列 | `collections.deque` | 头尾 O(1)，比 list 的 pop(0) O(n) 高效 |
| 约束 cells | `frozenset` | 不可变、可哈希，能放入 set 去重和 dict 索引 |
| 已访问记录 | `set` | O(1) 查重，避免约束推导指数膨胀 |

## 三、核心算法分析

### 3.1 游戏引擎：BFS 泛洪扩散

**问题**：掀开一个空格子（周围无雷）后，需要自动展开所有相邻安全区域，直到边界数字格为止。

**选型分析**：这是一个典型的区域填充问题。DFS 也能做，但递归深度在最坏情况下可达 O(rows × cols)，可能触发递归栈溢出。BFS 用队列迭代，无此问题。

**实现**：

```
将点击格入队
while 队列非空:
    取出队首
    若为数字格 → 不扩散（continue）
    若为空格 → 遍历 8 邻居：
        未翻/未标/非雷 → 掀开并入队
```

**复杂度**：时间 O(N)，空间 O(N)，N = 格子总数。每个格子最多入队一次。

**知识点**：BFS、队列（deque）、二维数组遍历

---

### 3.2 第一层：单点推理

**问题**：给定已翻开格子的数字和周围未翻开格子的集合，直接判断。

**选型分析**：这是最基础、最快的推理方式。只需要每格遍历一次，O(N) 扫描。不需要任何额外数据结构。

**实现**：

```
对每个已翻开数字格 (r, c)：
    covered = 周围 8 格中未翻且未标的集合
    remaining = grid[r][c].adjacent_mines - 周围已标雷数

    若 remaining == 0           → 全部安全
    若 remaining == len(covered) → 全是雷
```

**知识点**：二维数组遍历、集合运算、邻域搜索

---

### 3.3 第二层：约束方程组推理

**问题**：单点推理无法处理的局面，典型如 121 定式和边缘 11 定式。需要联合多个数字格的信息。

**选型分析**：这个问题本质上是**离散约束满足问题**。完整求解是 NP 完全的，但扫雷棋盘的约束有特殊结构——每个约束都来自一个数字格及其邻居，天然存在**子集包含关系**。利用这个性质，只需要 O(k²) 的成对比较即可完成推导，其中 k 是约束数量（通常远小于 N）。

**核心算法——约束子集减法**：

将每个数字格翻译为一条约束对象：

```
C = Constraint(cells={未翻邻居集合}, mines=剩余雷数)
```

对任意两个约束 C₁、C₂，若 C₁.cells ⊆ C₂.cells：

```
C_new.cells = C₂.cells - C₁.cells
C_new.mines = C₂.mines - C₁.mines
```

**实例**：

```
棋盘： 1 2 1     约束 C₁: {A,B} = 1
       A B C     约束 C₂: {A,B,C} = 2
                 约束 C₃: {B,C} = 1

C₁ ⊆ C₂ → C₂ - C₁ → {C} = 1 → C 是雷
C₃ ⊆ C₂ → C₂ - C₃ → {A} = 1 → A 是雷
标记 A、C 后 → B 安全
```

**为何不会指数爆炸**：每轮 deduce 只做 O(k²) 的子集判断。新生成的约束 cells 更小，迭代数次即收敛。加上 frozenset 去重，实际约束数远小于理论上限。

**知识点**：集合包含关系判断（issubset）、约束传播、等式消元

---

### 3.4 第三层：概率评估

**问题**：前两层都无法推进时，需要"猜"最安全的格子。

**选型分析**：精确计算每个格子的含雷概率需要枚举所有可能的雷区配置——组合数巨大，NP 完全。**蒙特卡洛采样**是工程上可行的近似方法：采样越多次，概率估计越准。

**实现**：

```
1. 构建当前约束集
2. 重复 samples 次：
   a. 随机初始化所有未翻格（偏置 = 剩余雷/未翻格数）
   b. 迭代修正：随机打乱约束顺序，逐条修正 diff
   c. 记录本次采样中每个格子是否为雷
3. 返回含雷频次最低的格子
```

**为何打乱约束顺序**：避免每次采样的修正过程产生系统性偏差，保证 500 次采样间的独立性。

**复杂度**：O(samples × iterations × k)，samples=500, iterations ≤ 100, k 为约束数。实际运行中通常在 0.1s 内完成。

**知识点**：蒙特卡洛采样、概率模型、随机化算法

## 四、算法复杂度总结

| 层 | 算法 | 时间复杂度 | 空间复杂度 |
|---|------|:---:|:---:|
| 引擎 | BFS 泛洪 | O(N) | O(N) |
| 第一层 | 单点推理 | O(N) | O(1) |
| 第二层 | 约束推导 | O(k²·d) | O(k²) |
| 第三层 | 蒙特卡洛 | O(S·I·k) | O(N) |

- N = 格子总数，k = 约束数，d = 推导轮次，S = 采样次数，I = 迭代次数
- 实际运行中 k << N，d ≤ 5，均在毫秒级完成
